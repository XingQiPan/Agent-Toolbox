# AI Toolbox Runtime 开发文档 v0.1

## 1. 项目概述

### 1.1 项目名称

暂定名：

```text
AI Toolbox Runtime
```

也可以后续改为：

```text
AITBX
Agent Toolbox
ToolBoxOS
Agent Tools Hub
```

### 1.2 项目定位

本项目是一个 **AI 可调用的工具箱运行时**，用于把大量 CLI、API、脚本、插件、MCP Server、Skill 工作流统一封装成 AI 可以安全调用的工具能力。

它不是传统网页工具集合，也不是单纯 MCP Server，也不是单一 Agent 框架。

核心目标是：

```text
让 AI 低 token、安全、可扩展地调用各种工具。
```

### 1.3 项目背景

传统工具箱，例如 MagicalBox 一类项目，主要面向人类用户，用户通过网页或 App 点选工具。我们的项目面向 AI Agent，重点不是“人打开网页点工具”，而是“AI 根据任务自动发现、选择、调用、组合工具”。

OpenAI 的 function calling/tool calling 支持用 JSON Schema 描述工具，让模型调用外部系统和应用能力；官方文档也提到，当应用有很多函数或大型 schema 时，可以结合 tool search 延迟加载不常用工具，避免一次性暴露所有工具定义。([OpenAI开发者][1])

MCP 的官方规范把服务器能力分为 Prompts、Resources、Tools 三类，其中 Tools 是模型控制的可执行函数，Resources 是上下文数据，Prompts 是交互模板；本项目会支持 MCP，但 MCP 只是适配层，不是内部核心协议。([Model Context Protocol][2])

Claude Agent Skills 的官方文档说明，Skill 由指令、元数据和可选资源组成，并采用按需加载、渐进式揭露的方式降低上下文成本；本项目可以借鉴这种思想，把复杂工作流做成 Skill Pack。([Claude平台][3])

### 1.4 参考项目定位

| 项目                         | 可借鉴点                                                         | 与本项目差异                       |
| -------------------------- | ------------------------------------------------------------ | ---------------------------- |
| MagicalBox                 | 工具分类、工具集合、轻量工具体验                                             | 面向人类点击使用，不是 AI-first         |
| Paperclip                  | Agent 组织架构、预算、治理、成本控制                                        | 更像 AI 公司/团队管理系统，不是工具运行时      |
| AionUi                     | 本地 Cowork、多 Agent、桌面 UI、统一界面                                 | 更偏 AI Agent 客户端，不是插件化工具执行平台  |
| Claude Code UI / Cloud CLI | 把 Claude Code、Cursor CLI、Codex、Gemini CLI 等 CLI 会话做成桌面/移动 UI | 更偏代码 CLI 前端，本项目面向通用工具箱和插件运行时 |

Paperclip 的公开描述强调它是一个用于“zero-human companies”的 Agent 编排系统，重点在组织结构、目标、预算、治理和 Agent 协作。([GitHub][4])

AionUi 的公开描述强调它是本地开源 Cowork 平台，让 AI Agent 在电脑上读文件、写代码、浏览网页、自动化任务，并让用户看到和控制 Agent 的行为。([GitHub][5])

Claude Code UI / Cloud CLI 的公开描述强调它是 Claude Code、Cursor CLI、Codex、Gemini CLI 等工具的桌面和移动 UI，可远程查看项目和会话。([GitHub][6])

## 2. 核心目标

### 2.1 一句话目标

```text
构建一个 API-first、CLI-first、Plugin-first 的 AI 工具箱运行时。
```

### 2.2 产品目标

1. 让 AI 可以通过 API 调用工具。
2. 让 AI 可以通过 CLI 调用工具。
3. 让开发者可以开发插件扩展工具能力。
4. 让工具能力可以导出为 MCP Server。
5. 让常用工作流可以导出为 Skill。
6. 降低工具调用时的 token 成本。
7. 保证工具调用安全、可审计、可限制权限。
8. 支持本地部署、私有部署、云端 API 三种形态。

### 2.3 非目标

第一版暂不做：

1. 不做完整 Agent 框架。
2. 不做复杂低代码工作流编辑器。
3. 不做大型插件市场。
4. 不做完整网页工具箱克隆。
5. 不做强依赖单一模型厂商的设计。
6. 不默认允许 AI 执行任意 shell。
7. 不默认允许插件访问用户全盘文件。

## 3. 设计原则

### 3.1 API-first

所有能力先通过 API 暴露，再考虑 UI。

### 3.2 CLI-first

每个工具都可以被 CLI 调用，方便 AI、脚本、CI/CD、本地开发者使用。

### 3.3 Plugin-first

工具不是写死在主程序里，而是通过插件注册。

### 3.4 Low-token

不能一次性把所有工具 schema 塞给模型。必须采用：

```text
工具搜索 -> 工具 schema 懒加载 -> 精准暴露少量工具 -> 执行
```

OpenAI 官方 tools 文档也说明，可以使用 function calling、tool search、remote MCP servers 等方式扩展模型能力，其中 tool search 用于运行时加载延迟工具定义。([OpenAI开发者][7])

### 3.5 安全优先

AI 调用工具必须经过权限、沙箱、审计、风险分级和人工确认。

OWASP LLM Top 10 把 Prompt Injection、Insecure Plugin Design、Excessive Agency 等列为关键风险；其中不安全插件设计可能导致远程代码执行，过度代理能力会带来隐私、可靠性和信任风险。([OWASP 基金会][8])

## 6. 核心模块

## 6.1 AI Tool Gateway

### 职责

1. 接收 AI 请求。
2. 根据用户意图搜索工具。
3. 动态加载工具 schema。
4. 与模型进行 tool call loop。
5. 调用 Tool Runtime。
6. 返回最终结果和 artifact。
7. 统计 token、成本、耗时。

### 关键能力

```text
toolbox.search_tools
toolbox.get_tool_schema
toolbox.run_tool
toolbox.get_file
toolbox.create_task
toolbox.get_task_status
```

### 推荐策略

第一阶段只暴露 3 个基础工具给模型：

```json
[
  "toolbox.search_tools",
  "toolbox.get_tool_schema",
  "toolbox.run_tool"
]
```

然后由模型先搜索工具，再加载少量相关 schema。

---

## 6.2 Plugin Registry

### 职责

1. 读取插件 manifest。
2. 注册工具。
3. 校验 schema。
4. 管理插件版本。
5. 管理插件权限。
6. 管理插件分类和标签。
7. 标记风险等级。

### 插件状态

```text
installed
enabled
disabled
quarantined
deprecated
uninstalled
```

### 工具风险等级

```text
low       纯计算、格式转换、只读操作
medium    写入临时文件、联网读取、OCR、文档转换
high      发邮件、写数据库、访问密钥、执行 shell
critical  删除数据、付款、提交代码、操作生产系统
```

---

## 6.3 Tool Runtime

### 职责

1. 执行工具。
2. 创建临时工作区。
3. 注入必要环境变量。
4. 传递 JSON 输入。
5. 接收 JSON 输出。
6. 管理超时、内存、CPU。
7. 处理 artifact 文件。
8. 校验输出 schema。

### 支持运行方式

```text
cli
http
python
node
container
wasm
```

### Runtime 优先级

MVP 阶段建议只实现：

```text
cli
http
python
```

第二阶段再实现：

```text
container
wasm
node
```

## 10. AI 调用流程

## 10.1 推荐工具调用流程

用户输入：

```text
把这个 PDF 拆成图片，压缩后打包成 zip。
```

流程：

```text
1. AI 收到用户请求
2. AI 调用 toolbox.search_tools
3. 系统返回候选工具：
   - pdf.to_images
   - image.compress_batch
   - archive.zip
4. AI 调用 toolbox.get_tool_schema 获取这几个工具 schema
5. Gateway 只暴露这几个工具给模型
6. AI 生成工具调用计划
7. Runtime 执行 pdf.to_images
8. Runtime 执行 image.compress_batch
9. Runtime 执行 archive.zip
10. AI 返回最终说明和 zip 文件
```

## 10.2 Provider Adapter 伪代码

```ts
async function runAgentTask(input: AgentInput) {
  const baseTools = [
    searchToolsSchema,
    getToolSchemaSchema,
    runToolSchema
  ];

  let messages = [
    systemMessage,
    userMessage(input)
  ];

  let availableTools = baseTools;

  while (true) {
    const response = await provider.createResponse({
      model: input.model,
      messages,
      tools: availableTools
    });

    if (response.type === "final") {
      return response.text;
    }

    for (const call of response.tool_calls) {
      if (call.name === "toolbox.search_tools") {
        const result = await searchTools(call.arguments.query);
        messages.push(toolResult(call.id, result));
      }

      if (call.name === "toolbox.get_tool_schema") {
        const schemas = await getToolSchemas(call.arguments.tool_names);
        availableTools = mergeTools(availableTools, schemas);
        messages.push(toolResult(call.id, { loaded: call.arguments.tool_names }));
      }

      if (call.name === "toolbox.run_tool") {
        const result = await runTool(call.arguments);
        messages.push(toolResult(call.id, result));
      }
    }
  }
}
```

## 10.3 Token 控制策略

### 规则 1：不暴露全量工具

错误做法：

```text
一次性把 200 个工具 schema 全部发给模型
```

正确做法：

```text
只暴露 search_tools / get_tool_schema / run_tool
```

### 规则 2：工具描述保持短

工具描述应该回答：

```text
这个工具什么时候用？
输入是什么？
输出是什么？
有什么风险？
```

不要写教程。

### 规则 3：工具输出不要太长

工具输出只返回：

```text
摘要
artifact id
关键统计
错误码
必要预览
```

不要返回完整文件内容。

### 规则 4：文件内容用 file_id 表示

不要把大文件内容塞进模型上下文。

### 规则 5：复杂任务用工具内部完成

例如：

```text
视频压缩
批量图片转换
OCR 大文件处理
表格分析
```

这些不要让模型参与每一步细节。

```text
MCP 适合跨客户端适配
但内部需要更强的权限、审计、文件、插件、成本管理
```

结论：

```text
内部使用自定义 Tool Runtime 协议
外部导出 MCP Server
```

---

### 25.2 工具执行必须结构化输出

原因：

```text
AI 不适合解析混乱 stdout
工具结果需要审计和复用
```

结论：

```text
所有工具必须输出 JSON
```

---

### 25.3 工具 schema 必须懒加载

原因：

```text
一次性暴露大量工具浪费 token
模型也更容易选错工具
```

结论：

```text
默认只暴露 search_tools / get_tool_schema / run_tool
```

---

### 25.4 高风险工具必须审批

原因：

```text
AI 可能误调用工具
外部内容可能诱导 AI 调用工具
```

结论：

```text
shell、邮件、数据库写入、删除、付款、提交代码必须审批
```

---

## 26. MVP 验收标准

MVP 完成时，应该可以做到：

```text
1. 启动本地 API 服务
2. 安装插件
3. 搜索工具
4. 查看工具 schema
5. 上传文件
6. 执行工具
7. 生成 artifact
8. 查看调用日志
9. AI 通过 API 调用工具
10. MCP 客户端可以调用 safe profile 工具
```

完整演示流程：

```bash
aitbx serve api
aitbx plugin install ./plugins/pdf-basic
aitbx plugin install ./plugins/image-basic
aitbx file upload ./demo.pdf
aitbx tool run pdf.extract_text --json '{"input_file_id":"file_xxx"}'
```

AI 演示：

```text
用户：把这个 PDF 提取文本，然后总结成 5 点。
AI：调用 toolbox.search_tools
AI：调用 pdf.extract_text
AI：返回总结
```

## 27. 推荐第一步

第一步不要先做 UI，也不要先做 100 个工具。

应该先做这个最小闭环：

```text
插件 manifest
工具注册
工具搜索
工具执行
文件 artifact
审计日志
AI Gateway
```

最小闭环命令：

```bash
aitbx plugin install ./plugins/json-basic
aitbx tool search "JSON 格式化"
aitbx tool run json.format --json '{"text":"{\"name\":\"aitbx\"}","indent":2}'
```

只要这个闭环跑通，后面 PDF、图片、视频、MCP、Skill、Web 控制台都可以自然扩展。

---

## 28. 最终架构结论

本项目最终应该采用：

```text
内部：AI Tool Runtime
外部：API + CLI + MCP + Skill
生态：Plugin SDK + Plugin Registry
安全：Policy Engine + Sandbox + Audit
成本：Tool Search + Schema Lazy Loading + Artifact ID
```

核心不是“工具很多”，而是：

```text
AI 可以可靠找到工具
AI 可以安全调用工具
工具可以低成本扩展
调用过程可以审计
插件生态可以成长
```

这就是这个项目的技术护城河。
